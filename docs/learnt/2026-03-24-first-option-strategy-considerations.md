# 首次所学：外部冻结归零不等于可以立即释放 STM bucket

## 触发问题

这次讨论的核心，不是“义务可用变成 0 后要不要 release”，而是账户表象相同的时候，策略内部语义可能完全不同。真正需要被记录下来的是：外部冻结已经覆盖义务，与 STM 自己仍有在途覆盖单或未吸收成交，这两类状态不能混写成一条规则。

## 场景拆分

### 场景 A：只有外部冻结

- 例子：义务方短仓 5，STM 自己没有挂单；人工另外挂了买平 5，于是 `short_frozen = 5`，可用义务变成 0。
- 账户侧表象：义务可用已经为 0。
- STM 自身状态：没有自己发出的覆盖单，没有自己待吸收的成交。
- 结论：STM 可以直接 release task / claim。
- 原因：策略内部没有未收尾的所有权，外部冻结已经把当前义务占满，继续占着 bucket 只会阻塞其他流程。

### 场景 B：STM 自己也有在途覆盖

- 例子：义务方短仓 5，STM 已经发了 5 手 SPECIAL 买开，但订单还没有到完全终态，或者成交还没被持仓同步吸收；与此同时，账户侧因为别的单也让可用义务看起来变成 0。
- 账户侧表象：义务可用同样为 0。
- STM 自身状态：仍有自己负责收尾的订单或成交。
- 结论：不能因为外部冻结把可用打到 0 就立刻 release。
- 原因：此时策略内部还有未完结的状态归属。如果 claim 提前释放，别的流程可能接管同一个 bucket，而 STM 自己那批订单和成交还在收尾，后续归属会变乱。

## 设计规则

- `available == 0` 只能说明账户快照层面的可用义务被占满，不能单独作为 release 条件。
- release task / claim / bucket 之前，必须同时确认：
  - 外部冻结已经覆盖当前义务，或者义务确实已被对冲掉；
  - STM 自有在途覆盖订单数量为 0；
  - STM 自有未吸收成交数量为 0；
  - 该 bucket 没有待收尾的生命周期动作。
- “是否可释放”是一个所有权判断，不是一个静态持仓判断。只看账户快照，会丢掉策略内部的在途语义。

## 应该显式跟踪的状态

- 外部冻结导致的义务占用量
- STM 自有在途覆盖挂单量
- STM 自有成交但尚未被持仓吸收的数量
- claim / bucket 当前是否仍归 STM 收尾
- 从“订单成交”到“持仓吸收完成”之间是否还有同步延迟

## 忽略后的风险

- 任务先释放，后续状态归属混乱，难以判断某笔成交到底该算在哪个 bucket 上。
- claim 提前释放后，别的流程可能立即接管这个 bucket，形成重复管理或相互覆盖。
- 已经发出的 STM 订单在收尾时找不到原始上下文，导致补账、撤单、对冲判断都变脆弱。

## 验证清单

- 回归场景 A：只有外部冻结时，确认可以安全 release。
- 回归场景 B：STM 自有在途覆盖存在时，即使账户侧可用义务为 0，也不得 release。
- 增补场景：STM 订单部分成交，但持仓尚未同步吸收，仍应保持 claim 占用。
- 增补场景：外部冻结撤掉后，系统能否重新识别需要继续由谁接管。

## 一句话结论

在期权策略里，“账户已经把义务占满”和“策略已经没有自己的收尾责任”不是一回事。前者是快照事实，后者才是 release claim / bucket 的业务前提。
